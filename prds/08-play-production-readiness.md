# Baskit PRD - Google Play Production Readiness

This checklist covers the release-gating work required before applying for Google Play production access, plus the code-hardening backlog that should be tracked before and after the first production rollout.

## Release Branch Discipline

- Current Play-ready release artifacts must come from the tagged release intended for upload.
- Do not merge large refactors into the currently uploaded Play build without cutting a new release version and uploading the new `.aab`.
- If the architecture refactor PR is merged before Play upload, cut a new patch release and use that release's artifacts instead of older artifacts.
- Keep release notes, `AppVersion`, `pubspec.yaml` version, Git tag, GitHub Release, and Play release metadata aligned.

## Code Gates

- `flutter analyze` passes on the release branch.
- `flutter test` passes on the release branch.
- Android release artifacts are signed with the release key, not debug signing.
- The Play Store release uses an `.aab` built from CI or a locally reproducible release build.
- Native debug symbols are generated and uploaded with the Play release.
- Firestore rules and indexes are deployed to the production Firebase project before Play review. Firestore rules are managed outside this repository and must be verified in that repo.
- Run a final clean checkout validation before tagging or uploading artifacts.

## Firebase, Sign-In, And App Integrity

- Add the Google Play App Signing SHA-1 and SHA-256 certificates to the Firebase Android app.
- Keep the upload-key SHA-1 and SHA-256 in Firebase if local or CI signed builds are used for testing.
- Download and commit an updated `google-services.json` after Firebase certificate changes.
- Verify Google Sign-In from a Play-distributed internal or closed testing build, not only from a locally installed APK.
- Verify guest-to-Google migration from a Play-distributed build with existing local lists.
- Consider Firebase App Check with Play Integrity:
  - Start in monitoring mode.
  - Review legitimate traffic and failure rates.
  - Enforce only after confidence is high enough to avoid locking out real users.

## Crashlytics, Privacy, And Data Safety

- Firebase Crashlytics is included, so Play Data Safety and the privacy policy must disclose crash diagnostics collection.
- Do not claim analytics collection unless an analytics SDK or custom analytics collection is added.
- Upload native debug symbols from CI for each Play release.
- Verify Crashlytics receives release-build crashes after the Play-distributed smoke test.
- Avoid logging personally identifiable information in Crashlytics keys, logs, and non-fatal reports.

## Play Console Setup

- Complete App Content forms:
  - Privacy Policy URL: `https://kmjayadeep.github.io/baskit/privacy-policy.html`
  - Account deletion URL: `https://kmjayadeep.github.io/baskit/delete-account.html`
  - Data Safety answers aligned with the current privacy policy and Crashlytics usage.
  - Target audience and content rating.
- Upload release notes matching the shipped version.
- Upload the release `.aab` and native debug symbols generated by CI.
- Review and resolve Google Play Pre-launch Report issues.
- Use a staged rollout for the first production release.

## Closed Testing Evidence

- Run the required closed test before applying for production access.
- Maintain at least 12 opted-in testers for at least 14 days if required for the developer account.
- Give testers a script that covers:
  - Guest list creation, editing, deletion, and offline usage.
  - Google Sign-In.
  - Local-to-cloud migration after sign-in.
  - Sharing by email.
  - Leave shared list.
  - Owner removes member.
  - Sign-out and return to guest mode.
  - Account deletion request path through the public web page.
- Collect concrete feedback, issue links, or screenshots from testers for the production-access application.

## Account Deletion Readiness

### Manual Account Deletion Process

- Monitor `contact@jayadeep.me` for deletion requests.
- Verify the requester controls the email address linked to the Baskit account.
- Delete the user's Firebase Auth account if it still exists.
- Delete `/users/{userId}`.
- Delete lists owned by the user unless the user requested export or transfer first.
- Remove the user from shared lists they do not own.
- Confirm completion by email within 30 days.

### Code Change Proposal

- Expose account deletion from Profile or About:
  - At minimum, add an in-app link to `https://kmjayadeep.github.io/baskit/delete-account.html`.
  - Prefer a clear signed-in account section with “Request account deletion” or “Delete account”.
- Harden `FirebaseAuthService.deleteAccount()` before relying on it for self-serve deletion:
  - Handle Firebase `requires-recent-login` by prompting re-authentication.
  - Do not clear local user data before remote account deletion succeeds.
  - Return a typed result or user-facing error message instead of only `bool`.
  - Ensure owned lists, memberships, and `/users/{userId}` cleanup are handled consistently by backend/manual process or Cloud Functions.

## Final Release Smoke Test

Use `prds/play-release-smoke-test-checklist.md` to record each Play-distributed release candidate smoke test.

- Install the Play-distributed build on a fresh device.
- Create a guest list and item.
- Sign in with Google and confirm the guest list migrates.
- Share a list with a second test account.
- Confirm both accounts see real-time list and item updates.
- Confirm leave-list and remove-member flows revoke access.
- Sign out and confirm the app returns to guest mode without exposing the signed-in user's cloud data.
- Confirm the About dialog opens the privacy policy.
- Confirm the app exposes the account deletion URL.
- Confirm Crashlytics diagnostics and debug symbols are available for the release.

## Production Code Improvement Backlog

These items improve resilience and maintainability. They are recommended before wider rollout, but large refactors should be released only after a fresh version bump, full validation, and Play-distributed smoke testing.

### 1. Account Management Hardening

- Add an obvious account deletion/request-deletion entry point in Profile or About.
- Improve deletion error handling and re-authentication flow.
- Ensure sign-out, deletion, and guest-mode transitions do not accidentally delete or expose the wrong data.

### 2. Crashlytics Robustness

- Wrap app startup in `runZonedGuarded` so uncaught async errors are recorded reliably.
- Keep `FlutterError.onError` and `PlatformDispatcher.instance.onError` configured for release builds.
- Record important caught service/repository failures as non-fatal Crashlytics events where useful.
- Avoid noisy reporting in expected offline/local-first paths.
- Do not include emails, list names, item names, or other PII in Crashlytics logs/keys.

### 3. Repository And Storage Simplification

- Continue simplifying after the storage architecture refactor:
  - Prefer `ShoppingRepository` providers in ViewModels.
  - Reduce direct `StorageService` usage in UI and feature code.
  - Keep `StorageService` only as a backwards-compatibility facade until callers are migrated.
  - Eventually remove the facade if all callers use repository abstractions directly.
- Preserve guest-first local storage, cloud routing, migration retry behavior, and share error messaging during each step.

### 4. Split Remaining Large Files

- Further split `FirestoreService` by concern:
  - list CRUD
  - item CRUD
  - members/sharing/permissions
  - user profile/contact lookup
  - migration helpers
- Continue extracting small widgets/helpers from list-detail UI where readability improves.
- Keep behavior-preserving refactors small enough to review and validate independently.

### 5. Production-Like Tests

- Add Firebase emulator or integration tests for:
  - Firestore permission behavior
  - sharing by email
  - local-to-cloud migration and retry after partial failure
  - leave-list and remove-member flows
  - account deletion cleanup expectations
- Add or maintain Firestore rules tests in the rules repository.
- Add widget tests for the Profile account deletion link/flow.
- Consider a smoke-test checklist artifact for each Play release.

### 6. Release Automation

- Consider automating Play upload via Fastlane or GitHub Actions.
- Keep a manual approval gate for production rollout.
- Archive `.aab`, APK, symbols, release notes, and validation logs for each production release.
