# Play Internal Release Automation

Baskit separates disposable **snapshot builds** from **internal release candidates** that may later be promoted to closed, open, or production testing. Snapshot builds are workflow artifacts only. Release candidates are tagged, signed, archived in GitHub Releases, and uploaded to the Google Play **internal** track when secrets are configured. Promotion to user-visible Play tracks remains manual in Play Console.

For the governing product/process requirements, see [`prds/14-release-candidate-and-changelog-strategy.md`](../prds/14-release-candidate-and-changelog-strategy.md).

## Release taxonomy

| Type | When to use | GitHub Release | Play upload | Release notes |
| --- | --- | --- | --- | --- |
| Snapshot build | Frequent maintainer/device smoke testing from `main`, `master`, or a manual workflow ref | No | No | Not required |
| Internal release candidate | A signed build that may be promoted beyond internal testing | Yes, marked prerelease | Internal track | Required, curated, cumulative since the last user-visible promotion |
| Closed/open/production promotion | Manual Play Console promotion of a selected candidate | Existing prerelease can be marked non-prerelease | Manual promotion | Same cumulative notes unless intentionally revised |

## Snapshot builds

Pushes to `main`/`master` continue to publish a short-lived `app-fast-debug-apk` artifact for maintainer testing. You can also run the `Build Flutter APK and App Bundle` workflow manually with `build_snapshot` enabled to create the same debug APK artifact from another ref.

Snapshot builds do **not** bump versions, create tags, create GitHub Releases, upload to Play, require `releases.json` entries, or change the promotion baseline. Use snapshots for disposable checkpoints.

## Internal release candidate workflow

1. Add curated user-facing highlights for the target version in `app/assets/whats_new/releases.json`.
2. Create the release candidate commit and tag with `./scripts/release.sh patch|minor|major`.
3. The `Build Flutter APK and App Bundle` workflow runs on the tag:
   - validates Flutter with `flutter analyze` and `flutter test`
   - builds signed release APK/AAB artifacts
   - exports cumulative Play release notes from the last promotion baseline
   - creates a GitHub Release marked as a prerelease
   - uploads the same AAB, cumulative notes, and native symbols to the Play internal track when `PLAY_SERVICE_ACCOUNT_JSON` is configured
4. Smoke test the Play-distributed internal build.
5. Promote the release manually from Play Console when ready.
6. After promotion is live, run `./scripts/mark_promoted.sh VERSION --track closed|open|production`, open a small PR for `docs/release-promotion-state.json`, and merge it after confirming the Play promotion.

## Required secret

Configure this GitHub Actions secret:

- `PLAY_SERVICE_ACCOUNT_JSON`: Google Play service account JSON with permission to edit internal-track releases for `com.cboxlab.baskit`.

The Play upload job does not need Android signing secrets. Signing remains in the release build job, which still requires the Android keystore secrets. The job reuses native debug symbols generated by the release build and passes `native-debug-symbols.zip` to Google Play with the upload action's `debugSymbols` input. Dart obfuscation symbols are still archived on the GitHub Release for manual crash deobfuscation.

## Release notes and promotion baseline

`app/assets/whats_new/releases.json` remains the canonical curated item catalog. `docs/release-promotion-state.json` records the latest version promoted to a user-visible Play track:

```json
{
  "lastUserVisibleVersion": "4.13.75",
  "track": "closed",
  "updatedAt": "2026-07-08"
}
```

The tag workflow exports Play notes in cumulative mode when this state file exists:

```bash
scripts/export_play_release_notes.py \
  --mode cumulative \
  --promotion-state docs/release-promotion-state.json
```

Cumulative mode selects releases where `lastUserVisibleVersion < release.version <= candidateVersion`, keeps only `userFacing=true` items, deduplicates related entries by `group`, prioritizes higher-impact and newer changes, caps the list with `--max-items`, and fails before upload if the rendered Play note exceeds the 500-character locale limit. Use `--mode single` when you need notes for only one semantic version.

## Validation logs

No separate validation-log archive is needed. The release workflow's `validate` and `release` jobs are the validation record, and the Play upload reuses the AAB built by that release workflow instead of rebuilding it.
